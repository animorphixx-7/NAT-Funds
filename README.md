## The problem

Most mutual fund platforms in India show you a star rating and a 1-year return. That's not analysis ‚ÄĒ that's a marketing number.

The metrics that actually matter when evaluating a fund ‚ÄĒ rolling returns across multiple windows, downside capture, Jensen's Alpha, Calmar ratio, Sortino ratio ‚ÄĒ are either buried in PDFs, hidden behind expensive platforms, or just not available to retail investors.

NAT Funds fixes that. It pulls live data from AMFI, NSE/BSE, and RBI, computes everything from raw NAV history, and serves it through a no-install browser UI. The entire backend ‚ÄĒ data pipelines, metric calculations, caching, API ‚ÄĒ is built from scratch in Node.js, without a framework doing the heavy lifting.

## What it computes

Everything in `services/metricsCalculator.js` is a pure function ‚ÄĒ no side effects, no hidden state. All inputs are NAV history arrays and TRI series. All outputs are numbers.

### Returns
| Metric | How |
|---|---|
| **CAGR (1Y / 3Y / 5Y / Inception)** | Closest-previous NAV lookup with a 7-day gap tolerance. Returns N.A. if history is too short ‚ÄĒ no fake numbers. |
| **Rolling Returns 1Y** | Daily-stepped windows across full NAV history. Reports min, max, and distribution. |
| **Rolling Returns 3Y** | Monthly-stepped (one window per calendar month). Reports p10 / p25 / p75 / p90 percentile bands. |

### Risk
| Metric | How |
|---|---|
| **Standard Deviation** | Annualised sample StdDev of monthly returns. 36-month lookback, minimum 12 months. |
| **Max Drawdown** | Two-pass O(n) scan ‚ÄĒ peak, trough, and recovery dates all tracked. Not just the number; the full timeline. |
| **SEBI Riskometer** | AMFI-published label when available, percentile rank of StdDev within sub-category otherwise. |

### Risk-adjusted
| Metric | How |
|---|---|
| **Sharpe Ratio** | `(avg_excess_return / ŌÉ_fund) √ó ‚ąö12`. Excess return computed using geometric monthly RFR from the live RBI 91-day T-bill rate. |
| **Sortino Ratio** | Same as Sharpe but denominator is downside deviation only ‚ÄĒ doesn't penalise upside volatility. |
| **Calmar Ratio** | `3Y CAGR% / |Max Drawdown%|`. Returns `null` if 3Y CAGR is unavailable or drawdown is zero. |

### Benchmark-relative
| Metric | How |
|---|---|
| **Beta** | Covariance / variance of 36 aligned monthly returns (fund vs real benchmark TRI). |
| **Jensen's Alpha** | `AnnFundReturn ‚ąí [Rf + ő≤ √ó (AnnBenchReturn ‚ąí Rf)]`. Geometric annualisation throughout. |
| **Information Ratio** | `(AnnFundReturn ‚ąí AnnBenchReturn) / TrackingError`. Requires ‚Č•36 aligned months. |
| **R¬≤** | Pearson correlation¬≤ between fund and benchmark monthly returns. 0‚Äď100 scale. |
| **Upside / Downside Capture** | Morningstar methodology. Requires ‚Č•10 up-months and ‚Č•6 down-months to be meaningful. |

### Peer scoring
| Metric | How |
|---|---|
| **Consistency Score (0‚Äď10)** | Composite: 30% median 3Y rolling ¬∑ 20% rolling positive % ¬∑ 20% downside capture (inverted) ¬∑ 20% Sortino ¬∑ 10% 3Y CAGR. Percentile-normalised within sub-category. |

### AMFI-published (sourced directly, not calculated)
TER, AUM (‚āĻ Cr), AMFI Information Ratios (1Y/3Y/5Y/10Y for Direct and Regular plans), SEBI Riskometer label, and fund-to-benchmark mappings are all pulled from official AMFI APIs and XLSX reports ‚ÄĒ not scraped, not approximated.

## Data sources

| Source | What it provides | When it refreshes |
|---|---|---|
| **AMFI NAV text feed** | Live NAV for ~9,100 schemes | Boot + on demand |
| **mfapi.in** | Full historical NAV per scheme | 24h disk cache per scheme |
| **AMFI Fund Performance API** | AUM, fund‚Üíbenchmark map, AMFI IR (1Y/3Y/5Y/10Y), SEBI Riskometer | Daily cron at 09:30 IST + `/admin/sync-aum` |
| **AMFI TER XLSX** | Total Expense Ratio per scheme | Daily cron at 09:00 IST + `/admin/sync-ter` |
| **NSE / BSE Indices API** | Benchmark TRI history (Beta, Alpha, IR, Capture) | Daily cron at 09:31 IST + `/admin/sync-tri` |
| **RBI / CCIL (91-day T-bill)** | Risk-free rate for Sharpe, Sortino, Jensen's Alpha | 24h in-memory cache |

### Caching ‚ÄĒ three tiers

The system uses a deliberate three-tier cache to keep restarts fast without burning API rate limits:

**Tier 1 ‚ÄĒ `data/*.json` (committed, long-lived):** TER, AUM, TRI, benchmarks, AMFI IR, and riskometers. Refreshed by scheduled cron jobs and admin endpoints. Benchmarks get a 90-day TTL specifically ‚ÄĒ they're semi-permanent under SEBI mandate and don't need daily re-fetching.

**Tier 2 ‚ÄĒ `cache/nav_<schemeCode>.json` (git-ignored, 24h TTL):** Per-scheme NAV history from mfapi.in. Auto-populated on first fetch, reused for 24 hours. On a fresh clone this folder is empty ‚ÄĒ it fills itself.

**Tier 3 ‚ÄĒ `cache/processed_funds.json` (git-ignored, 24h TTL):** The full computed fund list ‚ÄĒ every scheme with every metric applied. If this file is less than 24 hours old, the entire boot sequence (parse ‚Üí fetch ‚Üí compute) is skipped. A warm restart takes seconds instead of minutes.

## Boot sequence

What actually happens when you run `npm start`:

```
1. Load TER index from data/ter-data.json (sync from AMFI if missing)
2. Load AUM, Benchmark, AMFI IR, Riskometer from data/ stores
3. Fetch live RBI 91-day T-bill rate
4. Check cache/processed_funds.json age
   ‚Ēú‚ĒÄ‚ĒÄ < 24h old ‚Üí load and skip to step 8 (warm restart)
   ‚ĒĒ‚ĒÄ‚ĒÄ stale or missing ‚Üí continue
5. Parse live AMFI NAV feed (~9,100 schemes)
6. Batch-fetch historical NAV for top ~3,000 schemes (10 concurrent)
7. Compute all metrics in memory ‚Üí save processed_funds.json
8. Set dataReady = true ‚Üí begin serving
```

During steps 5‚Äď7, the frontend shows a live progress bar. It polls `/api/status` every second. No blank screen, no mystery spinner.

## Running it

### Prerequisites
- Node.js 18+
- Internet access on first boot (AMFI data fetch)

### Install and start

```bash
git clone https://github.com/harleenkhanuja/NAT-Funds.git
cd NAT-Funds
npm install

npm start       # production: node server.js
npm run dev     # development: nodemon with auto-restart
npm test        # 9 smoke tests, no network, ~0.5s
```

Open `http://localhost:3001` ‚ÄĒ the server serves the frontend automatically.

### Environment variables

| Variable | Default | Description |
|---|---|---|
| `PORT` | `3001` | HTTP port |
| `ADMIN_SECRET` | _(empty)_ | If set, `/admin/*` requires `X-Admin-Secret` header |
| `CORS_ORIGIN` | _(empty)_ | Explicit CORS origin if separating frontend from backend |
| `LOG_LEVEL` | `info` | Pino level: `trace` `debug` `info` `warn` `error` `fatal` |

## Design decisions worth explaining

**No frontend framework, no build step.** The SPA runs on plain HTML and ES modules loaded in strict dependency order. No Webpack, no Vite, no transpilation. This was deliberate ‚ÄĒ it removes an entire category of complexity, keeps the frontend debuggable in DevTools without source maps, and means a fresh clone just works.

**`metricsCalculator.js` is entirely pure functions.** ~1,200 lines, zero side effects. Every metric takes a NAV array and returns a number. This makes testing straightforward and makes it impossible for a metric computation to accidentally corrupt global state.

**The 90-day benchmark TTL.** AMFI publishes benchmark assignments per SEBI mandate and they almost never change. Hitting the API daily for something that changes quarterly wastes requests and introduces latency on sync operations. The 90-day TTL is a deliberate tradeoff.

**7-day gap tolerance on CAGR.** Indian markets have public holidays that create NAV gaps. A strict date lookup would produce N.A. for too many valid funds. The 7-day window finds the closest available NAV, which matches how AMFI itself calculates published returns.

**Warm restart design.** `cache/processed_funds.json` exists specifically so the server can restart during the day without re-running a 2‚Äď4 minute boot sequence. This is a real operational concern ‚ÄĒ if you're deploying on a small VPS and need to push a config change, you don't want users staring at a loading screen.

**Rate limiting at 200 req/min/IP.** The server aggregates from six external APIs. Exposing an uncapped public API that could trigger cascading upstream rate limit errors is a bad idea. The Express-layer limit is a backstop.

> **IDCW note:** Rolling returns for IDCW plans reflect NAV movement only ‚ÄĒ dividends paid out are excluded. This matches the methodology used by Tickertape and INDmoney.

## What I learned

The hardest part was not the metrics math ‚ÄĒ it was making the data pipeline reliable. AMFI's feed has inconsistent spacing. mfapi.in occasionally returns empty arrays for valid scheme codes. NSE's TRI API has different field names depending on the index. AUM data uses AMC names that don't match scheme names in the NAV feed, so `normaliseName()` in `scripts/` was a real project of its own ‚ÄĒ string normalisation, false-positive checks, match-rate audits across 9,100 entries.

The second hardest thing was the boot sequence. Getting from "server starts" to "data is ready to serve" in a way that's fast on warm restarts, correct on cold starts, and doesn't leave users confused about what's happening required building the three-tier cache system and the polling progress bar from scratch.

Everything else ‚ÄĒ the metrics, the API, the SPA routing ‚ÄĒ was straightforward once the data was clean.
